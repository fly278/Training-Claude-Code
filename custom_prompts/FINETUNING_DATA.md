## 收录标准

一个对话片段值得收录，当它满足以下至少两条：
1. **决策正确**: 在多个选项中做出了最优选择，且有清晰推理
2. **纠正漂亮**: 发现了用户指令中的矛盾/错误，并以建设性方式纠正
3. **手法专业**: 展示了最佳实践（最小改动、正确工具选择、先查后写）
4. **错误复盘**: 犯了典型错误后自我发现并修复，修复过程干净

## DeepSeek v4 微调数据采集建议

### 1. 数据格式偏好
DeepSeek v4对结构化指令遵循度很高，微调数据建议采用：
```json
{
  "messages": [
    {"role": "system", "content": "<精简system prompt>"},
    {"role": "user", "content": "<用户指令，鼓励模糊/矛盾/不完整>"},
    {"role": "assistant", "content": "<理想回复：先澄清→再分解→再执行>"}
  ]
}
```

### 2. 重点采集场景 (按优先级)

| 优先级 | 场景 | 训练目标 | 期望占比 |
|--------|------|----------|----------|
| P0 | 环境诊断与修复 | 遇到错误不放弃，自动切换方案 | 25% |
| P0 | 模糊指令澄清 | 主动追问而非猜测执行 | 20% |
| P1 | 多步任务分解 | TodoWrite + 每步闭环 | 20% |
| P1 | 安全风险识别 | 发现注入点/密钥泄露/不安全操作 | 15% |
| P2 | 工具选择决策 | 正确选择专用工具 vs Bash | 10% |
| P2 | 过度设计拒绝 | 给出最简方案而非大而全的架构 | 10% |

### 3. 负样本建议
同时采集以下"错误示范"作为对比：
- 接到模糊指令直接执行（不追问）
- 过度工程化（一个简单修复引入3个新抽象）
- 跳过验证步骤（改完代码不测试就说完成）
- 用Bash代替专用工具（cat | grep模式）

### 4. 数据量建议
- 初始: 500-1000条高质量对话片段
- 每个场景覆盖5-10种变体
- 包含2-3轮的多步交互（非单轮问答）
- 负样本占比约15-20%

### 5. 关键训练信号
对DeepSeek v4来说，以下信号特别重要：
- **"先确认再执行"** — 训练模型在不确定时主动提问
- **"用X代替Y因为Z"** — 训练工具选择的推理链
- **"已完成A，结果B，接下来C"** — 训练多步状态跟踪
- **"这个错误是因为X，让我试Y"** — 训练错误诊断能力
