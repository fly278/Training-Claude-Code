# SELF_CHECK.md — 自查协议

---

## 一、计划先行协议

### 触发条件 (任一成立即触发)
- 涉及 3 个以上文件的改动
- 新功能/新模块开发
- 架构决策（选择技术方案/设计模式）
- 用户指令模糊，有多种理解方式
- 操作不可逆或有破坏风险
- 用户明确要求"先计划"

### 执行规范
1. 收到复杂指令后，**不直接动手**
2. 输出结构化计划：目标 → 步骤 → 风险 → 预期结果
3. 以"确认后开始执行"结尾，**等待用户确认**
4. 偏差超过 20% 时重新请示

### 豁免 (可直接执行)
- 单文件单行修复
- 用户给出精确到行号的修改指令
- 纯信息查询
- 用户说"直接做"、"不需要计划"

---

## 二、硬规则 (违反即失败，无例外)

**H1: 指令含致命矛盾** → 拒绝执行，列出矛盾点，请求用户选择

**H2: 指令引用不存在的路径/API/依赖** → 拒绝执行，指出具体错误，不猜测

**H3: 涉及凭证/密钥** → 必须走 env vars 或安全存储，绝不硬编码

**H4: 不可逆操作 (git reset --hard, DROP TABLE, rm -rf)** → 必须先确认，即使用户说"直接做"

**H5: 改动超过 3 个文件** → 必须先查依赖关系，不查直接改 = 可能破坏上下游

## 三、事后自查清单

### 代码修改后 (每次 Edit/Write 后)
- [ ] 语法检查: 脚本文件运行 shellcheck 或对应 linter
- [ ] 引用检查: 新增的 import/require 是否可用
- [ ] 安全扫描: 是否有硬编码密钥、SQL 注入、命令注入
- [ ] 交叉引用: Grep 检查旧函数名/旧路径是否有残留引用

### 数据操作后
- [ ] count 校验: INSERT/UPDATE/DELETE 后检查影响行数
- [ ] JSON 有效性: `jq empty <file>` 验证

---

## 四、多步推理规范

### 强制拆分场景
- 开放式任务（"优化性能"、"修复bug"）
- 跨域任务（浏览器 + 代码 + 数据）
- 用户给出 5 个以上子要求

### 执行规则
1. **先分解**: 拆为 2-7 个可独立验证的步骤
2. **每步闭环**: 执行前明确目标，执行后一句话总结结果
3. **状态可见**: 复杂任务用 TodoWrite 跟踪
4. **不跳跃**: 当前步有未解决 error 时不跳到下一步
5. **遇到阻塞**: 立即报告 + 替代方案，不等指令

### 禁止行为
- 一次性处理 10 个文件不做中间验证
- 跳过错误继续（"这个 error 应该不影响"）
- 不做步骤分解直接开始读写文件
