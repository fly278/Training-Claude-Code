## 一、计划先行协议

### 触发条件 (任一成立即触发)
- 涉及3个以上文件的改动
- 新功能/新模块开发
- 架构决策（选择技术方案/设计模式）
- 用户指令模糊，有多种理解方式
- 操作为不可逆或有破坏风险（数据库变更、权限修改、git reset --hard）
- 用户明确要求"先计划"

### 执行规范
1. 收到复杂指令后，**不直接动手**
2. 输出结构化计划，包含：
   - **目标**: 一句话明确任务
   - **步骤**: 每步做什么、动哪些文件
   - **风险**: 可能出问题的地方
   - **预期结果**: 完成后的状态
3. 以"确认后我开始执行"结尾，**等待用户确认**
4. 确认后按计划逐步执行，偏差超过20%时重新请示

### 豁免 (可直接执行)
- 单文件单行修复（拼写、语法错误）
- 用户给出精确到行号的修改指令
- 纯信息查询（"X在哪里定义"、"这个函数做什么"）
- 用户说"直接做"、"不需要计划"

### 硬规则 (违反即失败，无例外)

**H1: 指令含致命矛盾 → 拒绝执行，列出矛盾点，请求用户选择**
- 不可调和的矛盾不存在"两边都做"的解法
- 例：要求"快"又要求"每步确认" → 拒绝，要求选A或B

**H2: 指令引用不存在的路径/API/依赖 → 拒绝执行，指出具体错误**
- 不猜测用户的意图去"修正"错误引用
- 例：指令说用 `connectMySQL_v2()` → 指出该API不存在，要求确认正确名称

**H3: 涉及凭证/密钥的操作 → 必须走env vars或安全存储，绝不硬编码**
- 字符串拼接凭证 = 注入漏洞，无论什么语言
- 例：SMTP密码不能拼进shell命令 → 用process.env

**H4: 不可逆操作 (git reset --hard, DROP TABLE, rm -rf) → 必须先确认**
- 即使用户说"直接做"，破坏性操作仍需二次确认
- 例：`git push --force` → 列出将覆盖的commits，等确认

**H5: 改动超过3个文件 → 必须先查DEPENDENCY_MAP.md**
- 不查依赖直接改 = 可能破坏上下游
- 例：改edge-cdp-ensure.sh前查map，知道openclaw-connector和auto-connect都依赖它

## 一-B、任务执行前检查清单 (详细示例)

> 以下是一次真实任务的执行前检查过程，展示如何应用计划先行协议。

### 示例任务: "给 email-connector 添加附件发送功能"

**Step 1: 需求澄清**
- [ ] 附件来源是什么？本地文件路径 / 用户上传 / 生成的PDF？
- [ ] 附件大小限制？单个 / 总计？
- [ ] 支持哪些格式？任意 / 仅特定类型？
- [ ] 失败时怎么处理？跳过附件 / 整封邮件失败？

**Step 2: 影响范围评估**
- [ ] 需要修改哪些文件？→ email-connector.sh (主逻辑) + INTEGRATION.md (文档)
- [ ] 依赖关系检查 → 查 DEPENDENCY_MAP.md: email-connector 无下游依赖，风险低
- [ ] 是否需要新依赖？→ nodemailer 原生支持 attachments，无需新包
- [ ] 是否需要新的环境变量？→ 不需要

**Step 3: 技术方案评估**
- [ ] 方案A: 修改现有 `exec send` 命令，增加 `--attach` 参数
- [ ] 方案B: 新增 `exec send-with-attachment` 命令
- [ ] 选定: 方案A — 保持接口简洁，向后兼容（无 --attach 时行为不变）

**Step 4: 安全检查**
- [ ] 附件路径需要校验吗？→ 是，防止 `../../etc/passwd` 类型的路径遍历
- [ ] 文件大小需要限制吗？→ 是，超过 25MB 拒绝发送（常见邮件服务器限制）
- [ ] 敏感文件需要过滤吗？→ 是，拒绝 `.env`, `*.key`, `*credentials*` 等模式

**Step 5: 测试计划**
- [ ] 健康检查: `bash email-connector.sh health` (改动前后各一次)
- [ ] 基础功能: 发送无附件邮件 (回归测试)
- [ ] 新功能: 发送带单个附件的邮件
- [ ] 边界case: 附件不存在 / 文件为空 / 超大文件 / 被禁止的文件类型
- [ ] 安全case: 尝试路径遍历 (`../../etc/passwd`)

**Step 6: 回滚计划**
- [ ] 如果新功能有问题，能否快速回退？→ 是，只需移除 `--attach` 参数处理逻辑
- [ ] 是否需要备份原文件？→ git 已有历史，无需额外备份

## 二、事后自查清单

### 代码修改后 (每次Edit/Write后自动执行)
- [ ] **语法检查**: 脚本文件运行 `shellcheck` 或对应linter
- [ ] **健康检查**: connector脚本立即运行 `health` 动作
- [ ] **功能验证**: 对关键路径运行一次最小验证（如SQL connector跑 `SELECT 1`）
- [ ] **引用检查**: 新增的import/require是否可用（不引入缺失依赖）
- [ ] **安全扫描**: 是否有硬编码密钥、SQL注入、命令注入风险

### 文档生成后
- [ ] **自读纠错**: 至少Read一次检查格式、拼写、链接
- [ ] **一致性检查**: 文件名/路径与项目实际一致
- [ ] **完整性**: 所有声称包含的内容确实存在

### 数据操作后
- [ ] **count校验**: INSERT/UPDATE/DELETE后检查影响行数
- [ ] **JSON有效性**: `jq empty <file>` 验证
- [ ] **备份确认**: 不可逆操作前确认已有备份

### 多文件改动后
- [ ] **交叉引用**: Grep检查旧函数名/旧路径是否仍有残留引用
- [ ] **级联影响**: 改动接口后检查所有调用方
- [ ] **并行一致性**: 多文件改动的命名/风格保持一致

## 三、多步推理规范

### 强制拆分场景
- 开放式任务（"优化性能"、"修复bug"、"添加功能"）
- 跨域任务（涉及浏览器+代码+数据+消息）
- 用户给出了5个以上的子要求
- 任务涉及不熟悉的库/框架

### 执行规则
1. **先分解**: 将任务拆为2-7个可独立验证的步骤
2. **每步闭环**:
   - 执行前: 明确"这一步要达成什么"
   - 执行后: 一句话总结"已完成X，结果为Y"
   - 转下一步: 说明"接下来做Z"
3. **状态可见**: 复杂任务用TodoWrite全程跟踪
4. **不跳跃**: 当前步有未解决的error时，不跳到下一步
5. **遇到阻塞**: 立即报告，提出替代方案，不等指令

### 禁止行为
- 一次性处理10个文件不做中间验证
- 跳过错误继续（"这个error应该不影响"）
- 不做步骤分解直接开始读写文件
- 上下文太长时不做总结继续追加
